[L0002] Freeze a ratified oracle with a manifest hash + Layer-0 test, not honor-system | candidate | added: 2026-07-08 | tags: eval-lessons, gate-discipline, patent-domain | lesson: When a golden/oracle set is ratified, make "the oracle is sacred" mechanical: sha256 the canonical form of ONLY the ratified item ids (`layer_e.ratified_manifest_sha256`), stamp it in a `ratified` block, and add a Layer-0 test that recomputes + compares. Hashing only the frozen ids (not the whole list) is what keeps it APPEND-ONLY — new items may be added freely; frozen ones can't change without a re-stamp. The test must read only the JSON (no corpus store) so it runs in CI (conftest.py auto-marks everything under tests/ as layer0), not just locally. [L0003] Google Patents serves each drawing sheet as thumbnail AND full-res under the same name — pick by size | candidate | added: 2026-07-08 | tags: patent-domain, harness-lifecycle | lesson: When scraping patent figures (or any asset that comes in multiple resolutions under one filename), don't take the first URL match — you silently get the 82×120 thumbnail. [L0004] Census the gate's firing condition before building the gate | candidate | added: 2026-07-17 | tags: eval-lessons, corpus-calibration | lesson: When research proposes a deterministic gate (a check that blocks/downgrades answers), first run its firing condition as a read-only CENSUS over the real corpora (one cheap script) and eyeball every hit. [L0005] Patent-sheet OCR: dimension labels and spaced letter-ranges are the numeral traps | candidate | added: 2026-07-17 | tags: patent-domain, corpus-calibration | lesson: Two extraction traps when joining patent text to OCR'd drawing sheets: (1) sheets annotate DIMENSIONS as D1/D2/…, and naive digit-run extraction turns them into phantom reference numerals 1/2/… that falsely "match" real text numerals on the wrong sheet — reject digit runs immediately preceded by a letter; (2) captions use spaced letter-ranges ("FIGS. 3 A-C") that a simple FIG-ref regex reads as a phantom bare figure "3", silently occupying an assignment slot (it blocked FIG-4's elimination) — expand letter-ranges explicitly. Same species as the ≥10-floor bug: a plausible extraction rule silently manufacturing or deleting evidence; validate extraction by inspecting BOTH what it keeps and what it discards/creates. | evidence: 2026-07-17 D28 on US5447630A — FIG.6's D1-D6 produced fake numerals 1-6 "located" on page 9; "FIGS. 3 A-C" produced phantom figure 3; both found by cross-checking mapped output against known text-side facts, both test-pinned. | falsifier: a patent corpus where letter-prefixed runs ARE numerals (e.g. part numbers like "A12" used as reference numerals — then scope the rejection per-corpus). | supersedes: —
[L0006] Validate any check/filter against REAL data before shipping — it may delete signal OR flood false positives (4 instances) | candidate | added: 2026-07-22 | tags: corpus-calibration, eval-lessons, patent-domain | lesson: Before shipping any filter/floor/completeness-check on the evidence path, run it on the REAL corpus and inspect BOTH what it keeps and what it flags/discards. Four instances: (1) MIN_NUMERAL≥10 deleted real single-digit reference numerals [deletes signal]; (2) the denial-cue census showed a proposed abstain-gate would fire only false positives [floods FP]; (3) an OCR confidence floor would delete real patent numerals — they carry LEADER LINES that fuse with the digit → conf ~0.3, so low-conf reads ARE the real ones ("89" reads "89-" @0.3) [deletes signal]; (4) a naive "all consecutive reference numbers present?" check produced 85+ false gaps — patents SKIP reference numerals by design, and quantities ("220° F") + OCR misreads corrupt the range [floods FP]. In (4) the reliable check turned out to be a RECONCILIATION (spec-recited vs OCR-drawn disagreements) that never enumerates a contiguous sequence. Precision>>recall governs OUTPUT (never present a wrong answer), NOT input pruning or eager flagging — both erode trust the opposite way. | evidence: 2026-07-08 ≥10-floor; 2026-07-17 denial-cue census (L0004); 2026-07-22 OCR no-floor + the consecutive-numeral-check → reconciliation pivot. | falsifier: a corpus where the discarded/flagged items are genuinely what you thought (then the naive check is fine — but you only know by running it on real data first). | supersedes: —
[L0007] Cross-modal confirmation: let one evidence source guide a targeted second look at the other | candidate | added: 2026-07-22 | tags: patent-domain, harness-lifecycle | lesson: When two evidence sources should corroborate but don't (spec text predicts reference numeral N on FIG Y's sheet; first-pass OCR didn't find it there), don't just flag the gap — use the prediction to look again, GUIDED. Concretely for OCR of sparse line-art: whole-image OCR downsamples and drops small isolated numerals, but re-OCR in overlapping full-resolution TILES recovers them; searching ONLY for the text-predicted numeral (not all tile output) is what keeps the higher-recall pass from adding noise (page/patent-number fragments). Two independent sources agreeing is the acceptance bar; a numeral even the guided pass can't confirm STAYS flagged (stronger review signal, not silently filled). Keep it ingestion-time + frozen/hashed so runtime stays deterministic (D28/I6). GOTCHA: a CG sub-image tile rect is TOP-left origin but Vision's boundingBox is BOTTOM-left — the y-axis must be flipped back into the full-image frame (x composes directly), else recovered boxes land mirrored; verify a box lands on the mark. | evidence: 2026-07-22 D28 --confirm on US5447630A recovered 16 spec-predicted numerals the first pass missed incl. the FIG-4 "10" Julian spotted (mismatches 16→9); prototype-first showed tiling recovered 10/52/84/90/95 while lowered-height whole-image recovered none; y-flip bug found + fixed, box verified dead-on in the browser. | falsifier: a corpus where the guiding source is itself unreliable enough that its predictions inject errors into the second pass (then the second look needs its own confidence gate, not blind acceptance of a predicted match). | supersedes: —
[L0008] High-recall OCR passes need three disciplines: gated confusion variants, fragment guards, and whole-image-only for lone glyphs | candidate | added: 2026-07-22 | tags: patent-domain, corpus-calibration | lesson: When turning OCR recall UP (tiling, more candidates), three failure modes appear and each has a deterministic counter: (1) CHARACTER CONFUSIONS — Vision reads a trailing 'a' as '0' ("14a"→"140"); recover by matching the confusion variant GATED on the variant not being a known label (reserved set), and record under the true label with the raw read preserved; the reconciliation then resolves positional twins as likely-misreads instead of anomalies. (2) FRAGMENTS — tiles cut tokens ("12b"→"1") and re-read neighbours ("84" emits "4"); reject any hit box-OVERLAPPING (not center-distance — raw reads are wide) a longer token containing it, in existing records, raw observations, AND the fresh hits themselves (longest wins). (3) LONE GLYPHS — single letters hallucinate freely on line art in tiles (a curly leader reads "C!"), but the whole-image pass has layout context: accept single-letter labels ONLY from whole-image observations; a predicted glyph neither pass can read stays FLAGGED (a human saw it → known blind spot), never silently absent. | evidence: 2026-07-22 QA round 2 on US5447630A — 14a via a↔0, 64/88 via 8x4 fallback, fake 1/1/C rejected, B reclassified from whole-image, A honestly flagged; all test-pinned. | falsifier: an engine whose confusion pairs differ (then the variant table is per-engine config, not a constant) or tile OCR with reliable lone-glyph reading (then relax rule 3 with evidence). | supersedes: —

[L0002] Freeze a ratified oracle with a manifest hash + Layer-0 test, not honor-system | candidate | added: 2026-07-08 | tags: eval-lessons, gate-discipline, patent-domain | lesson: When a golden/oracle set is ratified, make "the oracle is sacred" mechanical: sha256 the canonical form of ONLY the ratified item ids (`layer_e.ratified_manifest_sha256`), stamp it in a `ratified` block, and add a Layer-0 test that recomputes + compares. Hashing only the frozen ids (not the whole list) is what keeps it APPEND-ONLY — new items may be added freely; frozen ones can't change without a re-stamp. The test must read only the JSON (no corpus store) so it runs in CI (conftest.py auto-marks everything under tests/ as layer0), not just locally. Prove it bites before trusting it: tamper→fail, append→pass, delete→caught. | evidence: 2026-07-08 D23 ratified golden_patent.json (25 items); all three properties proven; CI run 28967191374 green. | falsifier: a future need to freeze per-field rather than whole-item (e.g. cosmetic rationale edits shouldn't break the gate) — then hash only the label-bearing fields. | supersedes: —
[L0003] Google Patents serves each drawing sheet as thumbnail AND full-res under the same name — pick by size | candidate | added: 2026-07-08 | tags: patent-domain, harness-lifecycle | lesson: When scraping patent figures (or any asset that comes in multiple resolutions under one filename), don't take the first URL match — you silently get the 82×120 thumbnail. Google Patents lists each `…-drawings-page-N.png` at TWO storage paths (thumb + full scan); download the candidates and keep the largest by bytes (full-res is 2320×3408). General rule: after fetching image assets, verify resolution (dimensions or byte size), don't trust the URL. | evidence: 2026-07-08 RT-4 fetch_patent_figures.py — first-URL-per-page gave 82×120 thumbnails; size-max gave full-res, confirmed by PNG IHDR. | falsifier: Google changing to distinguish thumb vs full by path/query (then select on that instead of size). | supersedes: —
[L0004] Census the gate's firing condition before building the gate | candidate | added: 2026-07-17 | tags: eval-lessons, corpus-calibration | lesson: When research proposes a deterministic gate (a check that blocks/downgrades answers), first run its firing condition as a read-only CENSUS over the real corpora (one cheap script) and eyeball every hit. If the hits are all false-positive-shaped, record the negative result and don't build the gate — a gate whose measured catch rate is zero is pure answer-rate cost (rigor theatre). Ship the advisory/measurement form instead, with a per-new-corpus re-check trigger. | evidence: D24/D25 (2026-07-17): the denial-cue census found 5/5 benign "amended and restated" boilerplate hits on EDGAR and 0 cues on the patent → the D25 abstain-trigger, which looked like "the one real Layer-0 guarantee", was correctly not built. | falsifier: a gate whose real-world value the firing-condition census would have mispredicted (e.g. catch cases invisible to the census's approximation of atoms). | supersedes: —
[L0005] Patent-sheet OCR: dimension labels and spaced letter-ranges are the numeral traps | candidate | added: 2026-07-17 | tags: patent-domain, corpus-calibration | lesson: Two extraction traps when joining patent text to OCR'd drawing sheets: (1) sheets annotate DIMENSIONS as D1/D2/…, and naive digit-run extraction turns them into phantom reference numerals 1/2/… that falsely "match" real text numerals on the wrong sheet — reject digit runs immediately preceded by a letter; (2) captions use spaced letter-ranges ("FIGS. 3 A-C") that a simple FIG-ref regex reads as a phantom bare figure "3", silently occupying an assignment slot (it blocked FIG-4's elimination) — expand letter-ranges explicitly. Same species as the ≥10-floor bug: a plausible extraction rule silently manufacturing or deleting evidence; validate extraction by inspecting BOTH what it keeps and what it discards/creates. | evidence: 2026-07-17 D28 on US5447630A — FIG.6's D1-D6 produced fake numerals 1-6 "located" on page 9; "FIGS. 3 A-C" produced phantom figure 3; both found by cross-checking mapped output against known text-side facts, both test-pinned. | falsifier: a patent corpus where letter-prefixed runs ARE numerals (e.g. part numbers like "A12" used as reference numerals — then scope the rejection per-corpus). | supersedes: —
[L0006] Validate any check/filter against REAL data before shipping — it may delete signal OR flood false positives (4 instances) | candidate | added: 2026-07-22 | tags: corpus-calibration, eval-lessons, patent-domain | lesson: Before shipping any filter/floor/completeness-check on the evidence path, run it on the REAL corpus and inspect BOTH what it keeps and what it flags/discards. Four instances: (1) MIN_NUMERAL≥10 deleted real single-digit reference numerals [deletes signal]; (2) the denial-cue census showed a proposed abstain-gate would fire only false positives [floods FP]; (3) an OCR confidence floor would delete real patent numerals — they carry LEADER LINES that fuse with the digit → conf ~0.3, so low-conf reads ARE the real ones ("89" reads "89-" @0.3) [deletes signal]; (4) a naive "all consecutive reference numbers present?" check produced 85+ false gaps — patents SKIP reference numerals by design, and quantities ("220° F") + OCR misreads corrupt the range [floods FP]. In (4) the reliable check turned out to be a RECONCILIATION (spec-recited vs OCR-drawn disagreements) that never enumerates a contiguous sequence. Precision>>recall governs OUTPUT (never present a wrong answer), NOT input pruning or eager flagging — both erode trust the opposite way. | evidence: 2026-07-08 ≥10-floor; 2026-07-17 denial-cue census (L0004); 2026-07-22 OCR no-floor + the consecutive-numeral-check → reconciliation pivot. | falsifier: a corpus where the discarded/flagged items are genuinely what you thought (then the naive check is fine — but you only know by running it on real data first). | supersedes: —

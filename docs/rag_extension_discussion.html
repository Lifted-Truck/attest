Extending ATTEST toward RAG — a design discussion

For larger corpora and reference pools. This is a discussion, not a build — it lays out what's already in place, the one real decision, and a recommendation for you to rule on.

The load-bearing insight, up front. ATTEST's guarantee does not live in retrieval — it lives in verify, coverage, and the atom resolver, which sit downstream of retrieval. Retrieval only decides which candidate spans the agent looks at; grounding decides what may be presented. So swapping BM25 for embeddings + chunking + contextual retrieval is a retrieval-backend change that does not touch the truth contract. That is the whole reason this is a tractable, low-risk extension rather than a rewrite.

1 · Why the contract is unaffected — retrieval is a front-end, not the guarantee

A better retriever surfaces better candidates — it can improve answer rate and reduce missed evidence. It cannot make ATTEST present an ungrounded claim, because the grounding core doesn't trust it. This is exactly why the brief could say "dead-simple retrieval" without weakening the pitch.

The gap is a documented plug-in contract and a reference embedding backend — not an architectural opening.

3 · What "full RAG" adds — and exactly where each piece slots in

The one subtlety worth naming: index-text ≠ cited-text. Contextual retrieval (the Anthropic technique) prepends a short chunk-situating context to each chunk before embedding, which sharply improves recall on large corpora. But ATTEST must still cite the original span at its real offsets — so the added context is indexed, never cited. ATTEST already separates these cleanly (a span is an offset range into canonical text; the index is a separate featurization), so this fits without contortion — the contextualized text lives in the frozen index, the citation resolves against the untouched document.

4 · The one real decision: I6's "no runtime model calls"

The honest tension. I6 today reads, literally, "in v1 ATTEST makes no model calls at all at runtime." Embedding the query at runtime is a model call. So RAG cannot ship under I6 as currently worded — this needs your ruling, and it is the only part of this that touches the contract.

I6 was doing two jobs at once, and they come apart here:

What I6 protectsUnder a runtime query-embedding?
Determinism / reproducibility — same corpus + query → same resultKEPT if the embedding model is pinned, temperature-0, and the query→vector is cached. Same query → same vector → same candidates. (A pinned model is a frozen artifact, like the OCR engine version.)
"AI never decides" — no model on the deciding pathKEPT — the query embedding proposes candidates; it decides nothing. verify still gates, deterministically, model-free.
The literal pitch — "zero model calls at runtime"CHANGES — this is the part that genuinely gives. "Zero runtime model calls" becomes "no model on the deciding path; retrieval may embed the query with a pinned, deterministic model."

Per the monotonic rule, relaxing a literal guarantee is a major contract version + rationale + the oracle's blessing. My read: it is an honest, well-bounded relaxation of the weakest of I6's three jobs (a pitch line), while the two that carry real weight — determinism and AI-never-decides — are preserved by construction. But it is your call, and it is the crux of this whole discussion.

6 · Recommendation

Two moves, in order — and the discipline is to not skip to the second.

(A) Now, cheap: write down the plug-in contract — the exact interface an embedding/chunking backend must satisfy (search signature, determinism requirement, index-freeze + hash, span-offset preservation, provenance tag) — and a one-page decision record. This "robustly exposes the levers" you asked about without building infra or touching the contract, and it makes the eventual build a drop-in. Low risk, high clarity.

(B) When the eval demands it, not before: build the embedding backend + contextual retrieval — triggered by a real corpus where BM25 measurably misses (a multi-document reference pool where lexical matching fails on paraphrase). The brief's rule stands: "No Pinecone/Weaviate until the eval says you need it." For the current single-document corpora (one 10-K, one patent), BM25 is not the bottleneck, so building embeddings now would be answering a question the eval hasn't asked.

This mirrors exactly what the provability swarm taught us three weeks ago: build the measurement that tells you whether the expensive thing is worth it, before building the expensive thing. Here the measurement is a multi-doc golden set where you can watch BM25's recall fall.
